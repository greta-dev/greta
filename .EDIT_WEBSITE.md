# Building and contributing to the greta website

This document explains how [the greta website](https://greta-dev.github.io/greta/)
is built, and how to edit its content or add an analysis case study or example
model. Before you submit a pull request, please
[open a GitHub issue](https://github.com/greta-dev/greta/issues/new) describing
the change and discuss it with the maintainers.

## Overview

The site is built with **standard [pkgdown](https://pkgdown.r-lib.org/)** using
the Bootstrap 5 template. There is no custom pkgdown fork any more — everything
is driven by:

- `_pkgdown.yml` — navbar, reference index, article grouping, and the bslib
  theme (greta's purple `#A379CC`, Roboto font).
- `pkgdown/extra.scss` — custom styling: the home-page hero banner, feature
  bullets, resource panel, and the code-block tint.
- `index.Rmd.orig` — the home page, including the hero banner HTML and the basic
  example.

The site is deployed automatically by GitHub Actions
(`.github/workflows/pkgdown.yaml`) to the `gh-pages` branch and served via
GitHub Pages. You do not deploy by hand.

## Building the website locally

From the top of the greta repository, in R:

```r
pkgdown::clean_site()
pkgdown::build_site()
# then open docs/index.html
```

`pkgdown::preview_site()` serves an already-built `docs/` in your browser; it
does not build anything itself.

For a quick look at just the chrome (navbar, theme, reference index) without
rendering articles:

```r
pkgdown::build_home()
pkgdown::build_reference()
```

## Pre-computed vignettes (IMPORTANT)

greta's vignettes load TensorFlow and run MCMC, which is far too expensive to
run on CI, on every `build_site()`, or during `R CMD check`. So the expensive
vignettes are **pre-computed**: the heavy R is run once, locally, and the
evaluated result is committed. Everything downstream then only runs pandoc over
already-evaluated markdown — no Python/TensorFlow required.

Each expensive vignette exists as two files:

| File | What it is | You edit it? |
|------|------------|--------------|
| `<name>.Rmd.orig` | The **source** — live `{r}` chunks that load greta and run MCMC. | **Yes** |
| `<name>.Rmd`      | The **generated** output — code with results and figures baked in. No live greta code. | **No — overwritten on re-render** |

The home page follows the same pattern: `index.Rmd.orig` → `index.md`
(pkgdown uses `index.md` in preference to `index.Rmd`).

The vignettes that are pre-computed are listed in `vignettes/precompute.R`.
Currently: `get_started`, `example_models`, `webpages/technical_details`,
`analyses/eight_schools`, `analyses/election_88`, and the home page. Static
vignettes (e.g. `webpages/why_greta`, `webpages/software`,
`webpages/contribute`) and cheap ones that are mostly `eval = FALSE`
(`installation`, `faq`) are left as ordinary `.Rmd` files.

### The workflow

Whenever you change an expensive vignette:

```
1. edit   vignettes/<name>.Rmd.orig          # always edit the .orig
2. run    Rscript vignettes/precompute.R      # the ONE step that needs greta/TF
3. commit the .orig + generated .Rmd/.md + figures  (all together)
```

`precompute.R` knits each `.Rmd.orig` to `.Rmd` (writing figures alongside) and
knits `index.Rmd.orig` to `index.md`. Run it from the repository root in an
environment where greta's Python stack works.

### Two rules that keep this correct

1. **Never hand-edit a generated `.Rmd`/`index.md`** — your edits are lost at the
   next `precompute.R`. Edit the `.orig`.
2. **Never set `cache = TRUE`** in a greta vignette. greta arrays wrap
   TensorFlow/reticulate external pointers that knitr cannot serialise, which
   fails with a cryptic `lazyLoadDBinsertValue ... subscript out of bounds`
   error. Pre-computing is the caching strategy; chunk caching is not needed.

### What ships and what doesn't

`.Rbuildignore` keeps byproducts out of the package tarball: `.Rmd.orig`
sources, `vignettes/precompute.R`, `index.md`, and `*_cache/` directories. The
committed generated `.Rmd` files are the real vignettes.

## Customising the look

- Theme colours and fonts: the `template.bslib` block in `_pkgdown.yml`.
- Everything else (hero banner, bullets, panels, code tint): `pkgdown/extra.scss`,
  which pkgdown compiles into the Bootstrap 5 theme automatically.
- The home-page hero markup lives in `index.Rmd.orig`; its images
  (`greta-header.png`, `main_icon.png`) come from `man/figures/` and are served
  from `reference/figures/` in the built site.

## Updating existing pages

Edit the relevant source and rebuild. For pre-computed vignettes, edit the
`.orig` and re-run `precompute.R` (see above).

To change ...        | Edit ...
---------------------|-------------------------------------------
Home page            | `index.Rmd.orig`
get started          | `vignettes/get_started.Rmd.orig`
technical details    | `vignettes/webpages/technical_details.Rmd.orig`
why greta            | `vignettes/webpages/why_greta.Rmd`
software             | `vignettes/webpages/software.Rmd`
contribute           | `vignettes/webpages/contribute.Rmd`

## Adding analysis case studies

Add an Rmarkdown document to `vignettes/analyses/`. If it uses greta (loads the
library or runs inference), add it as a **pre-computed** vignette: name the
source `<name>.Rmd.orig`, add `"analyses/<name>"` to the `heavy_vignettes`
vector in `vignettes/precompute.R`, and run the script.

Then add it to the `analyses` menu in the `navbar` section of `_pkgdown.yml`:

```yaml
- text: my analysis
  href: articles/analyses/my_analysis.html
```

## Adding example models

Example models show how to define greta arrays for a general type of model using
an example dataset. They should NOT contain calls to `model()`, or any plotting
or inference. Each example model is a self-contained Rmarkdown file in
`inst/examples/`, included as a child document from
`vignettes/example_models.Rmd.orig`. Because `example_models` is pre-computed,
edit the `.orig` and re-run `precompute.R` after adding an example.

Each example model file has:

- **title** — short, a level-3 header.
- **brief intro** — no more than a paragraph.
- **data section** (if needed) — a level-4 header `data` and a code chunk with
  `highlight = FALSE` and a unique chunk name ending in `_data`. Omit if the
  dataset ships with base R.
- **greta code section** — a level-4 header `greta code` and an R chunk defining
  the model, with a unique chunk name ending in `_greta`.
- **BUGS/JAGS code** (if relevant) — a level-4 header and a non-R markdown code
  block wrapped in `<div class="bugs">`.
- **Stan code** (if relevant) — a level-4 header and an R chunk (`echo = FALSE`,
  name ending in `_stan`) reading the equivalent model from the
  [Stan example models repo](https://github.com/stan-dev/example-models),
  wrapped in `<div class="stan">`.

Template:

````
### my example model

introduce the model here

#### data
```{r my_example_data, highlight = FALSE}
```

#### greta code
```{r my_example_greta}
```

### BUGS/JAGS code
<div class="bugs">
```
```
</div>

### Stan code
<div class="stan">
```{r my_example_stan, echo = FALSE}
cat(readLines('https://raw.githubusercontent.com/stan-dev/example-models/master/<path/to/model>.stan'), sep = '\n')
```
</div>
````

Then add a horizontal rule and a child chunk to
`vignettes/example_models.Rmd.orig`:

````
<hr>

```{r child='../inst/examples/my_example.Rmd'}
```
````

## Deployment

Pushing to `main` triggers `.github/workflows/pkgdown.yaml`, which builds the
site and deploys it to the `gh-pages` branch. This assumes the expensive
vignettes are pre-computed (so the runner never needs TensorFlow). If you change
an expensive vignette, remember to run `precompute.R` and commit the generated
files, or the deployed site will be stale.

## Common pitfalls

- **Raw HTML must be flush-left.** In `index.Rmd.orig` (and any vignette mixing
  raw HTML with markdown), do not indent the HTML tags. pandoc treats any line
  indented four or more spaces inside an HTML block as a code block, so indented
  `<div>`s render as literal text on the page. Keep every tag at column 0.
- **`NOT_CRAN` gating.** Some vignettes (e.g. `example_models`) only evaluate
  their greta chunks when `NOT_CRAN == "true"`. Child-document `child=` includes
  are pulled in *only when the chunk is evaluated*, so if this is unset the
  vignette renders empty. `precompute.R` sets `NOT_CRAN=true` for exactly this
  reason — don't remove it.

## Navbar logo (template override)

pkgdown's BS5 template does not put a logo in the navbar (it only shows the
package logo in content-page headers). The greta wordmark in the navbar
top-left comes from one small override, `pkgdown/templates/navbar.html`: a
verbatim copy of pkgdown's stock navbar template whose brand link shows the
wordmark image instead of the "greta" text.

The image is `pkgdown/assets/name_icon_on_white.png` (copied from
`logos/name_icon_on_white.png`). Everything in `pkgdown/assets/` is copied to
the site root at build time, and the template references it with pkgdown's
depth-correct `{{root}}` prefix so the path works on every page. `.navbar-logo`
in `pkgdown/extra.scss` sizes it.

To change the navbar logo, replace `pkgdown/assets/name_icon_on_white.png` (the
source wordmarks live in `logos/`). This is the only template override in the
site; when upgrading pkgdown, diff it against pkgdown's own
`BS5/templates/navbar.html` and re-apply the brand change if the stock template
has moved on.
